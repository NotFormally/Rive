# RiveHub i18n Guide

> **For agents and developers**: This is the single reference for how internationalization works in this project.

## Architecture Overview

```
messages/en.json          ← Source of truth (English)
    ↓ npm run i18n:sync
messages/{locale}.json    ← 58 auto-generated locale files
    ↓ next-intl
src/app/[locale]/...      ← Pages consume translations via hooks
```

**Stack**: [next-intl](https://next-intl-docs.vercel.app/) v4.8.3 on Next.js 16 (App Router)

---

## Key Files

| File | Role |
|------|------|
| `messages/en.json` | Source of truth — all English translations |
| `messages/*.json` | Auto-generated by sync script — **never edit manually** |
| `src/i18n/routing.ts` | Defines 59 locales + navigation helpers (`Link`, `redirect`, `useRouter`) |
| `src/i18n/request.ts` | Server-side config — dynamically imports locale JSON |
| `src/middleware.ts` | Locale detection: URL → cookie → Accept-Language → default `en` |
| `src/app/[locale]/layout.tsx` | Wraps app in `NextIntlClientProvider` with loaded messages |
| `scripts/i18n-sync.js` | Claude-powered auto-translation pipeline |
| `.i18n-cache/hashes.json` | MD5 hash cache for change detection (gitignored) |

## Namespace Convention

Each top-level key in `en.json` = **1 namespace** = 1 feature or page.

```
PascalCase naming:
  LandingPage     — Landing/marketing page
  Dashboard       — Main dashboard
  Sidebar         — Navigation sidebar
  Pricing         — Pricing page
  Auth            — Login/signup flows
  Common          — Shared strings (buttons, labels, errors)
  Reserve         — Inventory/reserve module
  HACCPBuilder    — HACCP form builder
  ...
```

**Rules:**
- One namespace per page or self-contained feature
- Use `Common` for strings shared across multiple pages (e.g., "Save", "Cancel", "Loading...")
- Keep keys flat within a namespace — avoid deep nesting
- Use `snake_case` for key names: `hero_title`, `btn_save`, `placeholder_search`

## How to Add Translations

### Step 1: Add keys to `en.json`

```json
{
  "MyFeature": {
    "title": "Feature Title",
    "description": "Some description",
    "btn_save": "Save",
    "placeholder_search": "Search..."
  }
}
```

### Step 2: Use in your component

**Client components** (most common):
```tsx
"use client";
import { useTranslations } from "next-intl";

export function MyFeature() {
  const t = useTranslations("MyFeature");

  return (
    <div>
      <h1>{t("title")}</h1>
      <p>{t("description")}</p>
      <button>{t("btn_save")}</button>
      <input placeholder={t("placeholder_search")} />
    </div>
  );
}
```

**Server components** (for metadata, SSR):
```tsx
import { getTranslations } from "next-intl/server";

export async function generateMetadata({ params }) {
  const { locale } = await params;
  const t = await getTranslations({ locale, namespace: "Meta" });
  return { title: t("page_title") };
}
```

**With interpolation** (dynamic values):
```tsx
// en.json: "welcome": "Hello {name}, you have {count} items"
t("welcome", { name: "Chef", count: 5 })
```

### Step 3: Sync translations

```bash
npm run i18n:sync        # Translates only changed/new keys
npm run i18n:sync:force  # Nuclear: retranslates everything (use sparingly)
npm run i18n:watch       # Live watcher for dev (auto-syncs on en.json save)
```

## Translation Sync Pipeline

**How it works:**

1. `en.json` is the single source of truth
2. The sync script computes MD5 hashes of each key's value
3. Compares against `.i18n-cache/hashes.json` to detect changes
4. Changed keys are chunked (50 keys per API call)
5. Claude Haiku translates each chunk to the target language
6. 5 languages are processed concurrently
7. Results are written to the locale's JSON file

**Translation rules enforced by the system prompt:**
- Brand names preserved: RiveHub, HACCP, POS, Toast, Square, Stripe, Claude
- Template placeholders preserved: `{count}`, `{name}`, `{percent}`, `{date}`
- HTML entities and emojis preserved
- Nautical metaphors culturally adapted (not literal)
- RTL languages (Arabic, Urdu, Farsi) maintain logical content order

**Important:** Never manually edit locale files other than `en.json`. They will be overwritten on next sync.

## i18n Checklist (for new pages)

Before shipping a page, verify:

- [ ] All user-facing text uses `t("key")` — no hardcoded strings
- [ ] Placeholders on inputs use `t("placeholder_xxx")` or `t("key")`
- [ ] Aria-labels on interactive elements use `t("aria_xxx")`
- [ ] Error messages use `t("error_xxx")`
- [ ] Page metadata uses `getTranslations` server-side
- [ ] All new keys are added to `en.json` under the correct namespace
- [ ] `npm run i18n:sync` has been run after adding keys
- [ ] Build succeeds: `npm run build`

## Routing

URL pattern: `/{locale}/{page}`

```
/en/dashboard        → English dashboard
/fr/pricing          → French pricing page
/ar/dashboard        → Arabic dashboard (RTL)
/zh-CN/dashboard     → Simplified Chinese
```

Navigation uses i18n-aware wrappers from `src/i18n/routing.ts`:
```tsx
import { Link, useRouter, usePathname, redirect } from "@/i18n/routing";

// These automatically preserve the current locale in URLs
<Link href="/dashboard">Go to Dashboard</Link>
```

## Common Patterns

### Loading states
```tsx
t("loading") // Add "loading": "Loading..." to your namespace
```

### Date/time formatting
Use the browser's `Intl` API with the current locale:
```tsx
new Date().toLocaleDateString(locale, { day: "numeric", month: "long" })
```

### Numeric placeholders
Purely numeric placeholders (e.g., `placeholder="20.00"`) do not need translation. Only translate text-based placeholders.

### Password field bullets
`placeholder="••••••••"` is a universal convention — no translation needed.

```bash
npm run i18n:lint    # Detect hardcoded text in TSX files
```

This script scans `src/app/` and `src/components/` for user-facing strings that aren't using `useTranslations`. Run it before PRs to catch missed strings.

## FAQ

**Q: I need to add a new language.**
A: Add the locale code to `src/i18n/routing.ts` locales array, add a mapping in `scripts/i18n-sync.js` LANG_MAP, then run `npm run i18n:sync`.

**Q: A translation is wrong.**
A: Fix it in the target locale file (e.g., `fr.json`). It will persist until the English source key changes. To force retranslation of specific keys, delete them from the cache in `.i18n-cache/hashes.json`.

**Q: Can I use nested keys?**
A: Keep it flat within namespaces. `"hero_title"` not `"hero": { "title": ... }`.

**Q: How do I know which namespace to use?**
A: Match the page or feature name. If the string is reused across pages, use `Common`.
